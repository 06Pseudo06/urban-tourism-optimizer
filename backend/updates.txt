---

## 3. Authentication System (Core Feature)
Implemented a complete user authentication pipeline:

### **Register**
* Accepts: `name`, `email`, `password`.
* **Bcrypt:** Hashes passwords before database insertion.
* Stores user records securely in MongoDB.

### **Login**
* Verifies credentials against hashed passwords.
* **JWT:** Generates a JSON Web Token for stateless authentication.

---

## 4. Security Layer
* Implemented JWT-based authentication middleware (`protect`).
* **Logic:**
  1. Extracts token from the request header.
  2. Verifies token validity.
  3. Attaches user object to the request (`req.user`).
  4. Blocks unauthorized access with standardized error handling.

---

## 5. Protected Routes
* Successfully deployed a protected endpoint: `/api/user/profile`.
* Verified that it is **only** accessible with a valid Bearer token.
* Confirmed the authentication pipeline is working end-to-end.

# Urban Tourism Optimizer — Itinerary Engine Upgrade Summary

## 🎯 Objective

Transform the system from a **basic place-based generator** into a **human-like intelligent travel planner** that produces realistic, diverse, and meaningful itineraries.

---

# 🔄 Evolution of the System

## Phase 1 — Basic System (Initial State)

* Used **Geoapify** for nearby places
* Generated itineraries by **sorting and filtering places**
* Issues:

  * Random parks and restaurants
  * No time structure
  * No travel realism

---

## Phase 2 — Data Enrichment Layer

* Integrated:

  * **Google Places API (New)** → ratings, reviews, images
  * **Distance Matrix API** → travel time
  * **Weather API (optional)**

* Added:

  * Place enrichment
  * Scoring based on rating + reviews

✅ Result:

* Better quality places
* More realistic scheduling

---

## Phase 3 — Structured Scheduling Engine

* Introduced:

  * **Time-based itinerary generation**
  * Arrival & departure times
  * Travel time between places

* Implemented:

  * Constraint-based scheduling
  * Opening hours + user time window

✅ Result:

* Realistic timeline
* No overlaps

---

## Phase 4 — Experience-Based Planning (Major Shift)

Changed approach from:

```text
Select places → Arrange
```

to:

```text
Design day → Fill with places
```

* Added:

  * Day structure (morning → evening → night)
  * Slot-based planning
  * Category balancing

✅ Result:

* Logical daily flow
* Reduced randomness

---

## Phase 5 — Place Quality Improvement

* Introduced **Google search-based discovery**:

  * “top tourist attractions”
  * “famous landmarks”

* Added:

  * Popularity scoring:

    ```text
    rating × log(review_count)
    ```
  * Filtering low-quality places

✅ Result:

* Inclusion of real tourist locations
* Removal of irrelevant places

---

## Phase 6 — Geographic Intelligence

* Implemented:

  * **Clustering based on location (lat/lng)**
  * Area-wise planning per day

* Improved:

  * Distance-aware sequencing
  * Reduced zig-zag travel

✅ Result:

* More efficient routes
* Real-world feasibility

---

## Phase 7 — Human Experience Layer (Final Refinement)

Added “human-like thinking”:

### Key Improvements:

* Experience-based flow:

  * Landmark → Activity → Relax → Food → Scenic

* Highlight moments:

  * Sunset spots
  * Scenic locations
  * Cultural anchors

* Diversity rules:

  * Max 2 food places/day
  * Avoid repeated categories
  * Mix of cultural, activity, and leisure

* Deduplication:

  * Removed duplicate places (e.g., CSMT vs full name)

* Relaxed over-optimization:

  * Allowed slightly longer travel for better experiences

✅ Result:

* Itinerary feels **natural and meaningful**
* Not just optimized, but enjoyable

---
